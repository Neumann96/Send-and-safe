# Send and Safe

Telegram Mini App для приватной передачи файлов с клиентским end-to-end
шифрованием. Сервер хранит только зашифрованные блоки и технические
метаданные: ключ файла, имя, MIME-тип и содержимое остаются на стороне
браузера отправителя и получателя.

## Что умеет

- загружает файлы до 512 МБ;
- шифрует содержимое, имя и MIME-тип файла в браузере через AES-256-GCM;
- режет файл на зашифрованные блоки по 4 МБ;
- хранит передачу максимум 48 часов;
- позволяет получить файл только один раз;
- поддерживает два режима доступа: секретная ссылка или ссылка плюс отдельный код;
- не требует регистрации и не передает Telegram ID на backend;
- хранит метаданные в SQLite, а зашифрованные блоки на локальном диске;
- отдает `/api/health` для проверки состояния контейнера.

## Как это работает

1. Отправитель выбирает файл и режим доступа.
2. Браузер создает случайный AES-256 ключ, шифрует метаданные и блоки файла.
3. Сервер получает только криптографический конверт, размер и ciphertext-блоки.
4. Получатель открывает ссылку, локально восстанавливает ключ и скачивает блоки.
5. После успешной расшифровки клиент подтверждает получение, и сервер удаляет запись
   SQLite и каталог с блоками.

Если получение не завершилось, данные все равно удаляются по TTL. Одноразовое
получение ограничивает повторные скачивания с сервера, но не может запретить
получателю копировать уже сохраненный файл.

## Локальный запуск

Требования:

- Go 1.23+
- Node.js 22+

```powershell
cd web
npm install
npm run build
cd ..
go run ./cmd/server
```

После запуска приложение доступно на `http://localhost:8080`.

Для разработки frontend:

```powershell
cd web
npm run dev
```

Vite проксирует `/api` на `http://localhost:8080`, поэтому backend должен быть
запущен отдельно.

## Docker

```powershell
docker compose up --build
```

Локальный compose публикует приложение на `http://localhost:8080`. SQLite и
зашифрованные блоки сохраняются в volume `send-and-safe-data`.

Production compose находится в `deploy/compose.production.yaml`: контейнер
слушает порт `8080`, наружу проброшен `127.0.0.1:8790`, данные лежат в
`/var/lib/send-and-safe`, а healthcheck обращается к `/api/health`.

## Конфигурация

| Переменная | Значение по умолчанию | Назначение |
| --- | --- | --- |
| `ADDR` | `:8080` | адрес HTTP-сервера |
| `DATA_DIR` | `./data` | каталог SQLite и зашифрованных блоков |
| `WEB_DIR` | `./web/dist` | собранный frontend |
| `MAX_FILE_BYTES` | `536870912` | максимальный размер одного файла, 512 МБ |
| `MAX_STORAGE_BYTES` | `8589934592` | общая квота активных передач, 8 ГБ |

TTL фиксирован в коде и равен 48 часам. Фоновая очистка запускается раз в
минуту и удаляет истекшие передачи.

## Модель безопасности

Ключ файла создается через Web Crypto API в браузере. Сервер видит:

- случайный ID передачи;
- размер файла и количество блоков;
- публичный криптографический конверт;
- SHA-256 хеши upload/download токенов;
- IP-адрес и время HTTP-запросов.

Сервер не видит:

- исходное имя файла;
- MIME-тип;
- содержимое файла;
- ключ расшифровки;
- код доступа в режиме `ссылка + код`.

### Секретная ссылка

```text
https://host/d/TRANSFER_ID#key=FILE_KEY
```

Ключ находится во fragment после `#`. Эта часть URL не отправляется HTTP-серверу,
но любой обладатель полной ссылки может расшифровать файл.

### Ссылка + код

В этом режиме ссылка не содержит ключ файла. Приложение генерирует код из
24 символов Base32, то есть 120 бит случайности. Через PBKDF2-SHA-256 с
310 000 итераций из кода выводится AES-ключ, которым защищается ключ файла.
На сервер отправляются только зашифрованный ключ и параметры KDF.

Ссылку и код лучше передавать разными каналами.

## API

Основные endpoints:

- `POST /api/transfers` - создать передачу и получить upload token;
- `PUT /api/transfers/{id}/chunks/{index}` - загрузить зашифрованный блок;
- `POST /api/transfers/{id}/complete` - отметить передачу готовой;
- `GET /api/transfers/{id}` - получить публичный manifest;
- `POST /api/transfers/{id}/claim` - зарезервировать одноразовое получение;
- `GET /api/transfers/{id}/chunks/{index}` - скачать блок с download token;
- `POST /api/transfers/{id}/consume` - удалить передачу после расшифровки;
- `DELETE /api/transfers/{id}` - удалить передачу по upload token;
- `GET /api/health` - проверить состояние сервиса.

## Telegram

Для публикации нужен HTTPS-домен и бот, созданный через BotFather. URL Mini App
должен указывать на корень приложения. Backend не использует `initData`, поэтому
Telegram-профиль пользователя не передается сервису.

Перед production-контейнером нужен reverse proxy с HTTPS, например nginx или
Caddy. В `deploy/` есть готовые примеры systemd и nginx-конфигурации для VPS.

Для привязки menu button к `@sendandsafe_bot` запустите:

```powershell
.\configure_bot.ps1
```

Скрипт запросит Bot API token скрытым вводом и отправит его напрямую только в
Telegram Bot API.


хранилище для зашифрованных блоков.
